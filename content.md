
## 摘要
本文介绍了一套基于**信息论与图论**的跨语言代码质量量化评价体系，旨在通过 **Rust 和 Tree-sitter** 引擎实现亚毫秒级的静态分析。该体系将《软件设计哲学》和《重构》等经典著作中的抽象概念转化为可计算的**熵值指标**，如物理-逻辑映射熵、状态空间扩张率和接口实现熵等。与传统工具相比，它更侧重于**架构整洁度**而非单纯的语法规范，能够精准识别模块耦合与生命周期错配。通过输出结构化的**诊断报告**，该系统可作为 AI 编码智能体的“架构教练”，提供确定性的重构反馈。最终目标是消除大模型的逻辑幻觉，确保人类与 AI 产出的工程制品具备高度的可维护性与内聚性。

## 1.模块抽象度
### 接口实现熵 (IIE) —— 量化“深模块”

- **概念来源：** John Ousterhout 认为，最好的模块应该像深不可测的海洋，表面平静（接口简单），内部复杂（实现强大）。
    
- **量化公式：**
    
    $$IIE = \frac{\text{Count}(\text{Public Signature AST Nodes})}{\text{Count}(\text{Internal Implementation AST Nodes})}$$
    
- **评价标准：** $IIE$ 越低越好。如果一个类的接口声明本身比它的内部实现还要复杂（$IIE > 1$），这就是一个典型的“浅模块”（Shallow Module），不仅没有降低系统的认知负荷，反而增加了 API 记忆成本。Agent 经常产出这类无意义的透传包装层。

## 2.逻辑内聚度
### 外部状态依恋度 (EAD) —— 量化“依恋情结”

- **概念来源：** Martin Fowler 提出的代码坏味道。指一个函数在计算时，过度依赖另一个类的属性。
    
- **量化算法 (基于数据流)：**
    
    1. 锁定目标函数 $F$。
        
    2. 统计 $F$ 读取自身所在词法作用域（或 `this`）属性的次数 $N_{self}$。
        
    3. 统计 $F$ 读取外部参数实体属性的次数 $N_{external}$。
        
- **量化公式：** $EAD = \max(0, N_{external} - N_{self})$
    
- **评价标准：** 当 $EAD$ 显著大于 $0$ 时，触发重构警报。系统将生成建议指令：“该函数的逻辑计算过度依赖外部对象，建议将此方法直接移动到被依赖的数据实体类中。”

## 3.修改波及效应
### 传递闭包变更半径 (TCR) —— 量化“霰弹式修改”

- **概念来源：** Martin Fowler 指出的一种架构腐化现象，即增加一个单一功能，却需要打开 5 个不同的文件进行修改。
    
- **量化算法 (基于图论深度优先搜索)：**
    
    1. 在全局引用图谱中，选中任意业务实体或模块 $M$。
        
    2. 计算从 $M$ 出发，能够影响到的所有下游节点的集合大小（Fan-out 传递闭包）。
        
- **评价标准：** TCR 的绝对值并不重要，关键在于**分布异常**。如果一个非核心工具类（如 `DateFormatter`）的 TCR 覆盖了整个项目 60% 的文件，说明架构层次被击穿。对于 AI Agent，如果在一次迭代中导致某模块的 TCR 激增，说明它引入了不当的全局耦合。

### AI正在烧我的token产出💩山？
最近在学着搞Harness engineering，目前看到很多人的工作流都只review测试，不review代码。但是我偶尔扫到一眼AI写出来的代码品味和我不一样，总有想去让它改改的冲动。

到底要不要review呢？如果代码全都是AI写了，根本不需要人动手了，那可以不关心代码质量了吗？

简单想象了一下，如果代码质量太差，腐化太快，同样的逻辑用更低效的表达方式，更多的行数，更绕的实现，AI读起来可能会浪费token，甚至可能后续迭代时，AI自己都改不动。

不过主要还是内心有一种冲动，想要实现让AI写出来的代码，质量和风格不会崩盘，最好品味还能跟我一样。

有一些品味倾向，用linter就可以检查，但最主要的还是那些更抽象的、跨语言的、设计层面的品味。

于是我想量化的这些更抽象的“品味”，让这个可计算的、量化的评价作为LLM评测自己产出结果的一环。先AI聊聊我的一些想法，让它检索看看有没有现成的方案可以满足我的需求。

在这里先给一个简单的总结。

借用信息论和认知心理学的概念，“信息熵越大，认知负荷越大”。

#### 信息熵：不确定性与预测成本

在信息论中，**熵（Entropy）**衡量的是系统的**不确定性**或**不可预测性**。

- **低熵代码：** 逻辑清晰、符合惯例（如标准的命名规范、设计模式）。当你看到函数名时，就能猜到它的实现。这种“可预测性”降低了你需要处理的新信息量。
    
- **高熵代码：** 充满意外。变量名误导、逻辑跳转随意（Spaghetti code）、隐式副作用多。每一行代码都可能跳出你预料之外的行为，这种**不确定性**迫使大脑必须维持高度警觉。
    
#### 认知负荷：看得好累

- **原生负荷（Intrinsic Load）：** 业务逻辑本身的复杂度（无法逃避）。
    
- **外在负荷（Extraneous Load）：** 由表达方式不当引起的额外负担。
    
    **高信息熵的代码增加了“外在负荷”**。因为代码分布越混乱，大脑在单位时间内需要接收和处理的非重复、无规律信号就越多。根据米勒定律，人类短时记忆只能处理 $7 \pm 2$ 个信息单元。当熵过高时，信息单元（Chunks）无法被有效压缩，大脑就会“宕机”。

### 场景1-相关指标：信息冗余度

于是gemini给了我一个公式，看起来很唬人其实就是数数然后除法：
##### 1. 信息冗余度指标：物理-逻辑映射熵 (PLME)

针对你提到的 `../../` 路径冗余问题，这本质上是**物理存储结构**与**逻辑引用结构**的过度耦合。

- **量化公式：**
    
    $$PLME = \frac{\sum_{i=1}^{n} \text{Distance}(Path_i)}{\text{Count}(\text{Unique Logical Modules})}$$
    
    - 其中 $\text{Distance}$ 是路径中 `../` 或目录层级的深度。
        
- **评价逻辑：**
    
    - 如果 PLME 很高，说明代码在用大量的物理细节（目录在哪）来表达简单的逻辑需求（我要用这个 Service）。
        
    - **Agent 检验点：** 观察 Agent 是否倾向于创建深层嵌套并硬编码路径。优秀的 Agent 应该倾向于配置逻辑别名（Alias）或使用统一的导出桶（Barrel files）。
        

#### 2. 依赖显式化程度：显式依赖率 (EDR)

针对你提到的“模块依赖不透明”问题。

- **量化指标：**
    
    $$EDR = \frac{\text{Defined Interfaces/Injections}}{\text{Total Module Interactions}}$$
    
- **量化方法：**
    
    - 通过 AST（抽象语法树）扫描，统计一个模块内“硬编码的单例/全局引用”与“通过构造函数/参数注入的引用”的比例。
        
- **评价标准：**
    
    - **低分：** 模块内部直接 `new` 另一个 Service，或者直接操作全局变量。
        
    - **高分：** 所有的依赖关系都通过依赖注入（DI）或配置表显式声明。
        
    - **对 Agent 的意义：** 很多 Coding Agent 喜欢写“自洽”但“封闭”的代码块。高 EDR 要求 Agent 产出的代码必须具备良好的**可接插性（Pluggability）**。
        

#### 3. 模块“提取成本”：传递闭包大小 (TCS)

针对你说的“复用一段代码需要手动收集所有依赖”。

- **量化指标：** 当我们选中模块 $A$ 时，其**最小可运行闭包**包含的节点数 $N$。
    
    $$TCS(A) = |\{B \mid A \to^* B\}|$$
    
    （$\to^*$ 表示逻辑上的依赖可达性）
    
- **评价维度：**
    
    - **独立性：** 如果 $TCS$ 占了整个项目的 50%，说明这个模块无法复用，它与整个项目是“血肉相连”的。
        
    - **Agent 局限性：** 这种指标能有效识别 Agent 是否在“制造面条代码”。Agent 往往为了完成局部任务，顺手引入了大量的全局依赖。
        

#### 4. 架构一致性偏差 (ACD)

这是一个衡量人类与 Agent 是否遵循既定“规矩”的指标。

- **量化方法：** 预设一组“禁止依赖”规则（例如：持久层不能引用 UI 层）。
    
- **计算：** $\text{ACD} = \text{违规连线数} / \text{总连线数}$。
    
- **工具实现：** 利用图数据库（如 Neo4j）存储代码的依赖图，运行 Cypher 查询来寻找“非法路径”。
